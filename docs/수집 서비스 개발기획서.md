# Trend Korea 수집 서비스 개발기획서 (통합본)
작성일: 2026-03-03  
버전: v1.0 (초안 PRD + 현행 코드 통합)

## 1. 문서 목적
- `docs/수집 로직 정리.md`의 제품 방향(PRD 초안)과 `src/utils`의 실제 수집/요약 로직을 하나의 실행 가능한 기획 문서로 통합한다.
- 현재 구현된 기능과 미구현 영역을 분리해, 운영 가능한 MVP 기준과 다음 개발 우선순위를 명확히 한다.

## 2. 서비스 목표
- 한국 뉴스 채널에서 빠르게 이슈 키워드를 추출한다.
- 키워드 기반으로 뉴스 본문/메타를 수집한다.
- 수집 결과를 키워드 단위로 요약해 API/피드/알림에 재사용 가능한 데이터로 저장한다.
- 장기적으로는 기사 단위가 아닌 이벤트 단위(이슈 타임라인) 서비스로 확장한다.

## 3. 범위 정의
### 3.1 현재 구현 범위 (코드 기준)
- 키워드 수집: `src/utils/keyword_crawler`
- 외부 뉴스 파이프라인 연동 수집: `src/utils/news_crawler`
- 네이버 뉴스 검색 API 수집: `src/utils/naver_news_crawler`
- LLM 기반 키워드별 뉴스 요약: `src/utils/news_summarizer`
- 전체 반복 오케스트레이션: `src/utils/pipeline`
- 별도 도메인 수집(생필품 가격): `src/utils/product_crawler`

### 3.2 PRD 기준 미구현/부분구현
- 이벤트 군집(Event Clustering): 미구현
- 진전 감지(Update Detection: NEW/MAJOR/MINOR/DUP): 미구현
- 속보 자동 판정(BREAKING_NEW/BREAKING_UPDATE): 미구현
- 이슈 타임라인 자동 누적/변경점(diff) 생성: 미구현
- 이슈 자동 종료/재오픈(cooldown/closed/reopen): 미구현
- 알림 큐/발송 파이프라인: 미구현
- Kafka 기반 분산 파이프라인: 미구현 (현재는 단일 오케스트레이터/배치 실행)

## 4. 현행 시스템 구조
## 4.1 모듈 구조
- `keyword_crawler`: 뉴스 채널 헤드라인 기반 키워드 추출
- `news_crawler`: 외부 `news-pipeline` subprocess 호출 래퍼
- `naver_news_crawler`: 네이버 검색 API 직접 호출
- `news_summarizer`: Ollama(OpenAI-compatible) 기반 통합 요약
- `pipeline.orchestrator`: 키워드 → 뉴스수집 → 요약 일괄 수행

## 4.2 실행 흐름 (현재)
1. 활성 뉴스 채널 조회 (`news_channel.is_active = true`)
2. 채널별 RSS/HTML에서 헤드라인 수집
3. 형태소 분석으로 통합 키워드 및 교집합 키워드 생성
4. 전략(`intersection` 또는 `aggregated`)으로 크롤링 키워드 선택
5. 외부 뉴스 파이프라인으로 기사 수집 (`run_news_crawl`)
6. 선택적으로 네이버 뉴스 API 기사 병합
7. 통합 기사 JSON을 LLM에 1회 요청해 키워드별 요약 생성
8. 사이클 결과(JSON) 및 요약/수집 DB 저장(옵션)

## 5. 상세 로직 설계 (현행 반영)
## 5.1 키워드 수집
- 진입점: `src/utils/keyword_crawler/crawler.py`
- 채널 소스: DB `NewsChannel` 활성 목록
- 수집 방식:
  - RSS URL이 있으면 RSS 우선 (`extract_headlines_from_rss`)
  - 없으면 HTML 파싱 (`extract_headlines`)
- 차단 감지: captcha/403/비정상 접근 패턴 문자열 기반 탐지
- 키워드 추출:
  - `kiwipiepy` 명사/고유명사 기반 명사구 추출
  - 하위 중복 구문 필터링
  - 결과: 채널별 키워드 + 전체 집계 키워드 + 교집합 키워드
- 저장:
  - `crawled_keywords` (채널/통합)
  - `keyword_intersections` (교집합)

## 5.2 뉴스 수집 (외부 파이프라인)
- 진입점: `src/utils/news_crawler/crawler.py`
- 실행 방식: `uv run news-pipeline ...` subprocess 호출
- 필수 설정: `NEWS_PIPELINE_DIR`
- 산출물:
  - 기사 JSON (`crawl.json`)
  - 리포트 JSON (`crawl_report.json`, 옵션)

## 5.3 네이버 뉴스 API 수집
- 진입점: `src/utils/naver_news_crawler/fetcher.py`
- API: `https://openapi.naver.com/v1/search/news.json`
- 인증: `NAVER_API_CLIENT`, `NAVER_API_CLIENT_SECRET`
- 변환 로직:
  - HTML 태그 제거(`<b>`)
  - 요약기 입력 포맷으로 변환 (`to_article_dicts`)
  - 기본 신뢰도 `confidence=0.8`
- 저장: `naver_news_articles` (모델명 `NaverNewsArticle`)

## 5.4 뉴스 요약
- 진입점: `src/utils/news_summarizer/summarizer.py`
- 모델 호출:
  - `OLLAMA_BASE_URL` (기본 `http://localhost:11434/v1`)
  - `OLLAMA_DEFAULT_MODEL` (기본 `gemma3:4b`)
- 특징:
  - 전체 키워드를 단일 프롬프트로 1회 호출
  - 비표준 JSON 응답 정규화(`_clean_json_response`)
  - 키워드 매핑: 정확매칭 → 부분매칭 → 순서매칭
- 저장:
  - 배치: `news_summary_batches`
  - 키워드 요약: `news_keyword_summaries`
  - 태그: `news_summary_tags`

## 5.5 파이프라인 오케스트레이터
- 진입점: `src/utils/pipeline/orchestrator.py`
- 기본값:
  - `repeat=10`
  - `top_n=30`
  - `max_keywords=5`
  - `limit=3`
- 키워드 선택 전략:
  - `intersection`: 교집합 우선, 부족분 aggregated 보충
  - `aggregated`: 빈도순 상위만 사용
- 결과 디렉토리: `cycle_outputs/run_<UTC timestamp>/cycle_##/`

## 5.6 별도 수집 도메인: 생필품 가격
- 진입점: `src/utils/product_crawler/fetcher.py`
- 대상: 한국소비자원 가격 정보 API
- 수집 대상:
  - 상품 마스터 (`ProductInfo`)
  - 상품 가격 이력 (`ProductPrice`)
- 특이사항:
  - API endpoint는 HTTP 사용 전제(코드 주석 명시)
  - JSON 시도 후 XML 폴백 파싱

## 5.7 동일 이슈키워드 뉴스 업데이트 기능 (신규 기획)
- 목적:
  - 같은 이슈키워드(예: `트럼프 관세`)에서 새 기사 유입 시 기존 이슈의 업데이트로 누적 관리한다.
  - 사용자에게는 "신규 이슈 생성"보다 "기존 이슈 업데이트"를 우선 노출한다.
- 동작 개요:
  1. 사이클 수집 시 키워드별 최신 기사 묶음 생성
  2. 키워드 정규화 후 기존 활성 이슈키워드와 매칭
  3. 동일 키워드 이슈 존재 시 신규 기사만 diff 계산 후 업데이트 추가
  4. 동일 키워드 이슈 미존재 시 신규 이슈 생성
  5. 업데이트 중요도(`MAJOR`/`MINOR`) 분류 후 정렬/알림 반영
- 동일 이슈키워드 판정 규칙(초기):
  - 1차: 정규화 키워드 exact match
  - 2차: 키워드 토큰 Jaccard 유사도 임계치 이상(예: 0.7+)
  - 3차: 핵심 엔티티(인물/기관/지역) 1개 이상 중복
- 기사 업데이트(diff) 판정:
  - URL/canonical URL 신규 여부
  - 제목 유사도가 낮고(중복 아님) 본문 핵심 문장/숫자 변화가 있으면 업데이트로 채택
  - 완전 중복은 `DUP` 처리해 이슈 카운트만 증가시키지 않음
- 상태 관리:
  - `issue_keyword_state.status`: `active`/`cooldown`/`closed`
  - `last_seen_at` 기준 N시간(초기 48h) 무업데이트 시 `cooldown`, 재유입 시 재활성화
- 결과:
  - 같은 키워드 이슈의 업데이트 타임라인 누적
  - 최신 업데이트 시각/중요도 기준 피드 정렬
  - MAJOR 업데이트만 알림 전송

## 9. 갭 분석 (PRD vs 현재 코드)
## 9.1 강점
- 실제 운영 가능한 수집-요약 배치 파이프라인이 이미 동작 가능한 상태
- CLI와 DB 저장 경로가 분리되어 개발/운영 전환이 쉬움
- 교집합 키워드 전략으로 탐지 품질 개선 기반이 마련됨

## 9.2 핵심 갭
- 이벤트 모델 부재로 "같은 사건 묶기"와 "진전 감지" 불가
- 동일 이슈키워드 업데이트 누적/재노출 로직 부재
- 속보 전용 자동 판정 레이어(룰+모델 하이브리드) 부재
- 타임라인 업데이트의 "이전 대비 변경점(diff summary)" 자동 생성 부재
- 이슈 생명주기 자동화(종료/재오픈) 부재
- 기사 중복 제거(Dedup) 체계 부재
- 알림/개인화 기능 미구현
- 아키텍처가 배치 중심이라 실시간성(분 단위) 제어가 제한적

## 10. 단계별 개발 계획
## 10.1 Phase 1: 현행 파이프라인 안정화 (1~2주)
- 목표:
  - 실패 복구 가능한 실행 단위 정리
  - 수집량/실패율/지연 모니터링 지표 확보
- 작업:
  - 각 단계 결과 스키마 검증
  - 에러코드/재시도 정책 표준화
  - 사이클 실행 리포트 DB 적재

## 10.2 Phase 2: 기사 표준화 + 중복 제거 (2~3주)
- 목표:
  - `raw_articles` 중심 데이터 레이어 확보
  - URL/제목/시간창 기반 중복 판정
- 작업:
  - canonical URL 정규화
  - 중복 플래그 및 대표 기사 선정 규칙 도입

## 10.3 Phase 3: 이벤트 군집 + 업데이트 감지 (3~4주)
- 목표:
  - 이벤트 단위 피드 구성
  - `NEW/MAJOR/MINOR/DUP` 라벨 도입
- 작업:
  - `events/event_articles/event_updates` 모델 구현
  - `issue_keyword_state/issue_keyword_alias` 모델 구현
  - 동일 키워드 매칭 + 기사 diff 계산기 구현
  - 속보 자동 판정 레이어 구현(`BREAKING_NEW`, `BREAKING_UPDATE`)
  - 룰 기반 트리거 + 수치변화 감지

## 10.4 Phase 4: 사용자 기능 (2~3주)
- 목표:
  - 키워드 구독 및 MAJOR 알림 제공
- 작업:
  - `keywords/keyword_matches/notifications` 구현
  - 알림 채널 1종(이메일 또는 푸시) 우선 연동

## 10.5 Phase 5: 무인 운영 자동화 (2~3주)
- 목표:
  - 휴먼 터치 없이 속보/타임라인 품질 유지
- 작업:
  - 타임라인 변경점(diff summary) 자동 생성
  - 이슈 자동 종료/재오픈 정책 구현
  - 품질 게이트 자동화(JSON 검증, 요약 실패 fallback, 중복 폭주 차단)
  - 운영 이상징후 자동 알림(수집량 급감, 소스 장애, 모델 실패율 급등)

## 15. 구현 가이드 (휴먼 터치 없는 자동화 기준)
### 15.1 파이프라인 단계 (권장)
1. `Ingest`: 키워드 수집 + 뉴스수집 + 네이버수집 결과를 `raw_articles`에 통합 적재
2. `Normalize`: canonical URL, 발행시각, 채널/출처 표준화
3. `Dedup`: exact + near duplicate 판정(`duplicate_of_article_id`)
4. `Match`: 기존 active issue와 키워드/엔티티/유사도로 매칭
5. `Classify`: `BREAKING_NEW`, `BREAKING_UPDATE`, `MAJOR`, `MINOR`, `DUP`
6. `Timeline`: `event_updates` append + `diff_summary` 생성
7. `Rank`: live/top 점수 계산 후 `live_feed_items`, `issue_rank_snapshots` 반영
8. `Notify`: `user_alert_rules` 조건 충족 시 발송 큐 적재

### 15.2 백엔드 구현 방식
- 워커 실행:
  - 현재 `src/utils/pipeline/orchestrator.py`를 유지하고 step 함수를 세분화
  - 장기적으로는 스케줄러/APScheduler 또는 큐 워커로 분리
- 배치 주기:
  - 기본 60초(속보), 5분(요약/랭킹 재계산), 1시간(정리/종료판정)
- 멱등성:
  - 기사 멱등키: `canonical_url` + `published_at bucket`
  - 업데이트 멱등키: `issue_id + article_hash`
- 실패 복구:
  - 단계별 상태 컬럼(`pending/running/success/failed`) + retry_count
  - 요약 실패 시 fallback 텍스트(제목 기반 1문장) 저장

### 15.3 API 설계 추가안
- `GET /api/v1/feed/live`
  - 최근 속보 스트림, cursor pagination, `type` 필터(`breaking|major|all`)
- `GET /api/v1/feed/top`
  - Top Stories N건, 점수/업데이트 수/최근 업데이트 시각 제공
- `GET /api/v1/issues/{issue_id}/timeline`
  - 기존 trigger + `diff_summary`, `update_type`, `sources` 포함
- `POST /api/v1/users/me/alert-rules`
  - 관심 키워드/중요도/시간대/빈도 제한 설정

### 15.4 랭킹/판정 규칙 (초기 룰)
- Breaking Score:
  - 최근성 가중치 + 출처 신뢰도 + 수치 변화 유무 + 트리거 단어
- Top Score:
  - `recent_updates*0.4 + saved_count*0.2 + tracked_count*0.2 + source_weight*0.2`
- Auto Close:
  - `last_seen_at` 48h 경과 -> `cooldown`
  - `cooldown` 72h 추가 경과 -> `closed`
  - 재유입 시 `reignited`

### 15.5 최소 구현 순서 (2주)
1. DB: `raw_articles`, `event_updates`, `issue_keyword_state`, `live_feed_items`
2. 워커: dedup + issue match + update classify
3. API: `/feed/live`, `/issues/{id}/timeline`
4. 알림: MAJOR/BREAKING_UPDATE만 발송
5. 운영지표: 속보지연, 중복억제율, 업데이트연결율, 알림CTR

## 16. 신규/업데이트 판정 강화안 (중요)
### 16.1 설계 원칙
- 단순 이진 판정(`신규`/`업데이트`) 대신 점수 기반 다단계 판정으로 운영한다.
- 판정 순서는 반드시 `DUP 제거 -> 이슈 후보 매칭 -> 업데이트 점수 분류` 순으로 고정한다.

### 16.2 처리 순서
1. `DUP 1차 필터`: canonical URL exact match
2. `DUP 2차 필터`: 제목 SimHash/본문 fingerprint near-duplicate
3. `후보 이슈 검색`: 최근 72시간 `active/cooldown` 이슈 상위 K개
4. `후보 점수 계산`: 키워드/엔티티/임베딩/시간 근접성 종합
5. `분류`: `NEW` / `MINOR_UPDATE` / `MAJOR_UPDATE` / `DUP`

### 16.3 후보 이슈 검색 규칙
- 후보 범위:
  - `issue_keyword_state.status in (active, cooldown)`
  - `last_seen_at >= now - 72h`
- 검색 키:
  - 정규화 키워드 exact
  - 토큰 유사도(Jaccard)
  - 엔티티 overlap(인물/기관/지역)
- 선택:
  - 점수 상위 `K=5` 이슈만 상세 판정

### 16.4 업데이트 점수식 (초기 버전)
- `update_score = 0.35*semantic_similarity + 0.20*entity_overlap + 0.20*time_proximity + 0.15*keyword_overlap + 0.10*source_weight - duplicate_penalty`
- 주요 피처:
  - `semantic_similarity`: 제목+본문 임베딩 유사도
  - `entity_overlap`: 핵심 엔티티 교집합 비율
  - `time_proximity`: 직전 업데이트와의 시간 거리 점수
  - `keyword_overlap`: 정규화 키워드 토큰 중복률
  - `source_weight`: 출처 신뢰도(공식/통신사 우대)
  - `duplicate_penalty`: 제목/본문 near-duplicate 강도

### 16.5 라벨 임계값 (초기 권장)
- `DUP`: DUP 필터 통과 실패 또는 `duplicate_penalty`가 임계치 초과
- `NEW`: 후보 최고 점수 `< 0.45`
- `MINOR_UPDATE`: `0.45 <= score < 0.70`
- `MAJOR_UPDATE`: `score >= 0.70` 이고 아래 major 조건 1개 이상 충족

### 16.6 MAJOR 조건 (변경점 기반)
- 수치 변화:
  - 인명/금액/규모/시간 등 숫자 값이 기존 대비 유의하게 변경
- 상태 변화:
  - 예: `검토 -> 확정`, `의혹 -> 공식발표`, `수사착수 -> 기소`
- 주체 변화:
  - 새로운 핵심 기관/인물 등장
- 신뢰도 조건:
  - 저신뢰 출처 단독으로는 MAJOR 승격 금지

### 16.7 단기 재전송 억제 규칙
- 동일 이슈에서 10~20분 내 고유정보 없는 유사 기사 다수 유입 시 1건만 타임라인 반영
- 나머지는 `DUP` 처리하여 노출/알림 억제

### 16.8 저장 스키마 확장 (권장 컬럼)
- `event_updates`:
  - `update_score` (float)
  - `update_type` (`NEW|MINOR|MAJOR|DUP`)
  - `major_reasons` (jsonb: 숫자변화/상태변화/주체변화)
  - `matched_issue_id`
  - `duplicate_of_article_id`
- `raw_articles`:
  - `semantic_hash`, `title_hash`, `canonical_url`
  - `entity_json`, `normalized_keywords`

### 16.9 품질 검증/튜닝 루프
- 로그 적재:
  - `features`, `predicted_label`, `final_label`, `feedback(optional)`
- 주기:
  - 주 1회 임계값/가중치 재조정
- 목표:
  - 업데이트 연결율 85%+
  - DUP 억제율 70%+
  - MAJOR 정밀도 80%+
