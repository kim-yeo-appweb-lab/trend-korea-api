## 5. 핵심 기능 정의

### 5.1 수집(Ingestion)

- 입력: RSS, 공개 뉴스 검색 API, 공식 발표 페이지(보도자료), 재난/안전 공지(선택)
- 출력: RawArticle(원문/메타) 이벤트 파이프라인으로 전달

**요구사항**

- 폴링 주기: 소스 특성에 따라 30초~5분(기본 1~3분)
- 실패/재시도: 지수 백오프 + 회로차단
- 소스별 QoS: rate limit 준수

### 5.2 정규화/본문 추출(Normalisation)

- URL 정규화(utm 제거, canonical)
- 언론사/발행시각/섹션/기자 등 표준화
- 본문 추출(boilerplate 제거)
- 언어: 한국어 중심(영문 소스는 옵션)

### 5.3 중복 제거(Dedup)

- 1차: URL exact match
- 2차: 제목+발행사+시간창(예: ±2시간) 기준 유사도
- 3차: SimHash/MinHash(본문) 또는 임베딩 유사도

**출력**

- `duplicate_of_article_id` 또는 `is_duplicate` 플래그
- 원문은 보존하되, 프론트 노출은 대표 1건

### 5.4 이벤트 군집(Event Clustering)

- 이벤트 = 동일 사건/이슈를 설명하는 기사 묶음
- 온라인 매칭: 새 기사 들어올 때 기존 이벤트에 붙일지 결정

**기준(초기)**

- 제목/본문 키워드 overlap
- 엔티티(인물/기관/지역) overlap
- 임베딩 유사도(선택) + 최근성 가중치

### 5.5 진전 감지(Update Detection) — 핵심

이벤트에 새 기사가 추가될 때, “정말로 업데이트인가?” 점수화.

**룰 기반(초기)**

- 트리거 단어: `발표/확인/공식/착수/중단/재개/구속/기소/체포/사망/부상/피해/규모/복구/경보` 등
- 숫자 변화 감지: 피해액/인명/규모/시간/수치
- 출처 가중치: 공식/통신사/주요지 > 기타
- 시간 간격: 마지막 major update 이후 Δt

**라벨**

- `NEW_EVENT`: 새 이벤트 생성
- `MAJOR_UPDATE`: 핵심 사실 변화/공식 발표/수치 확정
- `MINOR_UPDATE`: 관련 반응/부가 설명
- `DUPLICATE`: 실질 정보 동일

### 5.6 요약(Summarisation)

- 이벤트 대표 요약: “현재까지 핵심 3줄 + 최신 진전 1줄”
- 업데이트 요약: 각 업데이트는 1~2문장

**주의**

- 저작권/약관 리스크 최소화:
  - 원문 재배포가 문제가 될 수 있으므로, **짧은 요약 + 출처 링크** 중심
  - 가능하면 “인용/재가공” 범위를 내부 정책으로 제한

### 5.7 검색/필터/알림

- 필터: 분야, 지역, 시간, 출처, “업데이트만”
- 관심 키워드: 키워드/기관/지역/기업 단위
- 알림 조건:
  - 키워드 매칭 + `MAJOR_UPDATE` 일 때만
  - 야간 알림 제한(옵션)

## 8. 시스템 아키텍처(권장)

> Kafka 기반 파이프라인 설계

### 8.1 파이프라인(토픽)

1. `ingest.raw` : 수집기에서 RawArticle 발행
2. `process.normalise` : 정규화/본문추출
3. `process.dedupe` : 중복판정
4. `process.cluster` : 이벤트 매칭/생성
5. `process.update_detect` : 진전 감지 + event_updates 생성
6. `publish.feed` : 프론트/알림에 노출할 피드 이벤트
7. `notify.queue` : 알림 발송 워커

### 8.2 서비스 구성

- **Ingestor**(Node/Python): 소스별 수집, rate limit 관리
- **Extractor**: 본문 추출/정규화
- **Deduper**: SimHash/MinHash/임베딩
- **Clusterer**: 이벤트 군집 로직
- **Summariser**: 이벤트/업데이트 요약
- **API Server**(FastAPI/NestJS): 피드/검색/상세/키워드/알림
- **Worker**: 알림 발송
- **DB**: Postgres(+pgvector 옵션)
- **Cache**: Redis
- **Observability**: Prometheus/Grafana, Sentry

## 9. 개발 방법(구체적 단계)

### 9.1 1단계: 수집기 + Raw 저장

- RSS 5~20개로 시작(속보/사회/정치/경제)
- `raw_articles`에 저장 + canonical_url unique로 중복 1차 방지
- 간단한 관리자 스크립트로 소스 on/off

### 9.2 2단계: 정규화/본문 추출

- trafilatura/readability 계열로 body_clean 생성
- 실패 시 fallback(제목+리드만)

### 9.3 3단계: 중복 제거

- 제목 유사도 + 시간창 + SimHash(본문) 적용
- 대표 기사 선정 규칙: (출처 점수, 본문 길이, 최신성)

### 9.4 4단계: 이벤트 군집(초기 룰)

- 최근 24~72시간 이벤트만 후보로 조회
- `relation_score` 계산 후 threshold 넘으면 attach, 아니면 new event
- 이벤트 타이틀: 대표 기사 제목을 정규화(불필요한 접두어 제거)

### 9.5 5단계: 진전 감지(룰 기반)

- 트리거 사전 구축 + 숫자 변화 감지
- `MAJOR_UPDATE` 발생 시 `last_major_update_at` 갱신
- 프론트 상단은 major 업데이트 기준으로 정렬

### 9.6 6단계: 요약/알림

- 이벤트 요약은 “대표 기사 + 최신 major” 기반
- 키워드 매칭은 이벤트 summary + title + entities 기준
- 알림은 major일 때만(초기 노이즈 방지)
