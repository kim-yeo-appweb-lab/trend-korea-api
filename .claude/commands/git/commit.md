# Commit

변경사항을 분석하여 Conventional Commits 형식으로 커밋을 생성한다.

## 절차

1. `git status`와 `git diff`로 전체 변경사항 파악
2. 변경사항을 **가장 작은 논리적 단위**로 분리
3. 각 단위별로 관련 파일만 `git add`하여 개별 커밋 생성
4. 커밋 완료 후 `git status`로 결과 확인

## 분리 기준

하나의 커밋에는 하나의 목적만 담는다:

- 새 파일 추가 vs 기존 파일 수정 → 분리
- 기능 코드 vs 설정/문서 → 분리
- 서로 다른 도메인 변경 → 분리
- 리팩토링 vs 기능 변경 → 분리

분리가 애매하면 **쪼개는 쪽**을 선택한다.

## 커밋 메시지 형식

```
{type}: {subject}

{body}
```

### 타입

- `feat` — 새로운 기능
- `fix` — 버그 수정
- `refactor` — 코드 리팩토링
- `style` — 코드 포맷팅 (동작 변경 없음)
- `docs` — 문서 변경
- `test` — 테스트 추가/수정
- `chore` — 빌드, 설정 변경
- `perf` — 성능 개선
- `migration` — DB 마이그레이션

### 규칙

- 제목은 한국어, 50자 이내
- 이모지 사용 금지
- body는 선택사항이며, 필요시 "왜" 변경했는지 설명
- `git add -A` 또는 `git add .` 사용 금지 — 파일을 명시적으로 지정
